# What's already been done: IFC/BIM -> Minecraft prior art

## 1. Direct IFC/BIM -> Minecraft attempts

The space is real but thin: a handful of named, end-to-end pipelines exist, but **none is an open, scriptable, IFC-native voxelizer that emits modern Minecraft schematics** — which is roughly what this project already has.

- **FME (Safe Software), IFC/Revit -> Minecraft** — [community.safe.com/data-7/ifc-revit-to-minecraft-2198](https://community.safe.com/data-7/ifc-revit-to-minecraft-2198). The most concrete *named, end-to-end* IFC-to-Minecraft pipeline. FME models Minecraft as a regularly-spaced point cloud (each point = one block), reads IFC/Revit, voxelizes, and writes a world; an `AttributeValueMapper` maps source material/category numbers to block IDs. Commercial/closed. Confirmed live. *Caveat:* the exact attribute/raster-band layout (Band0=Z, Band1=blockID, etc.) comes from KB snippets that 403'd — treat as reported, not verified.
- **Revit -> Minecraft via octree voxelization (danimosite)** — [danimosite.wordpress.com/2021/06/27/revit-minecraft](https://danimosite.wordpress.com/2021/06/27/revit-minecraft/). Working C# Revit-API code: `GetModelExtents()`, structural-element filtering, recursive octree (~depth 7), leaf-box vs. geometry via `ElementIntersectsSolidFilter`, emits `DirectShape` voxels. A clean *technique* precedent; the octree + solid-intersection approach transfers to IFC if you swap in the IfcOpenShell iterator. Confirmed live.
- **VASA — Voxel-based Architectural Space Analysis (Rhys Goldstein, Autodesk)** — [revit.news/2021/09/introducing-vasa](https://www.revit.news/2021/09/introducing-vasa-a-tool-for-voxel-based-architectural-space-analysis/). Maintained Dynamo package that voxelizes Revit/STL into a uniform grid "inspired by Minecraft," then runs pathfinding/daylighting/visibility. Same mesh->grid primitive, but analysis-oriented and Revit/Dynamo-bound — not a Minecraft exporter.
- **BeIMCraft (Timothy Hegarty, Ulster University)** — [beimcraft.com](https://www.beimcraft.com/). A Minecraft *mod* injecting BIM/construction workflow concepts for AEC education (foundations-first, stability/H&S/cost). Real and well-documented (the earlier "DNS failure" was transient). It is domain precedent and evidence that IFC/Revit-into-Minecraft for AEC is an established use case — not a reusable converter.

**Takeaway:** the closest direct precedents are either commercial/closed (FME), authoring-tool-bound (octree-in-Revit, VASA), or an educational mod (BeIMCraft). An open Python IFC->`.schem`/`.litematic` pipeline like this one is *not* well-covered prior art.

## 2. Real-world geo/building -> Minecraft precedents (proof at scale + lessons)

These prove voxelizing authoritative spatial data into Minecraft works at country and city scale, and they surface the recurring pitfalls a building pipeline will also hit.

- **Denmark in Minecraft (Danish Geodata Agency / SDFE, 2014)** — [geoboxers.com/denmark-in-minecraft](https://geoboxers.com/denmark-in-minecraft/). The canonical "whole country, official geodata, 1:1" build (~300k dataset downloads). The team founded GeoBoxers. Not downloadable since 2020. Confirmed.
- **GeoBoxers (WorldBloxer / GeoCraft)** — [geoboxers.com/from-point-cloud-to-minecraft](https://geoboxers.com/from-point-cloud-to-minecraft/). Most transferable to BIM: explicitly ingests architectural/BIM and LiDAR/point-cloud data. Surfaces two pitfalls directly relevant here — **(1)** color quantization to ~16 colored blocks; **(2)** noisy/non-homogeneous source needing statistical cleanup (analogous to messy IFC tessellations).
- **City of Helsinki — Minecraft-Helsinki3D+** + open-source [City-of-Helsinki/mesh_to_schematic](https://github.com/City-of-Helsinki/mesh_to_schematic). Building-scale (1 block = 1 m) city from a municipal reality mesh; the GitHub tool is an *inspectable* OBJ -> voxelize -> `.schematic` reference chaining CloudCompare/MagicaVoxel/FileToVox. MIT, but **archived read-only 2025-10-06**. Documents real operational pitfalls: long single-machine runtimes, **resumability**, manual base-world assembly. This is the closest open analogue to the mesh->schematic stage.
- **Ordnance Survey — Minecraft GB** — [theregister.com/2025/04/12](https://www.theregister.com/2025/04/12/ordnance_survey_releases_blocky_britain/). National-scale flagship; the 2025 build is 100B+ blocks (Jonathan Allsup) with a 2014 83B-block Guinness predecessor. Demonstrates the **block-resolution-vs-fidelity** tradeoff (25 m/block at country scale vs 1 m only at city scale) and the **recognizability problem** (thin/small features blur at coarse resolution). *Confidence note:* the 2013-v1 / 22-billion / 25-m-per-block and OS-Terrain-50/VectorMap source-data details are likely true but **not supported by the cited Register URL** — cite cautiously.
- **Blockholm (Stockholm, Sweco Position, 2013)** — [fme.safe.com/blog/2013/10](https://fme.safe.com/blog/2013/10/real-world-minecraft-meet-blockholm-sweden/). FME-driven GIS->Minecraft at 1:1; deliberately shipped terrain/parcels but left **buildings empty/crowdsourced**. Relevant to deciding which IFC layers to bake vs. leave editable, and to using a spatial DB (PostGIS) to manage a large voxel world.
- **Lantmäteriet — Sweden in Minecraft** — [svenskbyggtidning.se/2020/04/28](https://www.svenskbyggtidning.se/2020/04/28/utforska-sverige-med-minecraft/). National-scale; explicitly notes including **thin linear features (roads/rail)** is the hard differentiator — the direct analogue to special-casing thin BIM elements (walls, mullions, beams, railings).
- **Voxel Earth** — [voxelearth.org](https://voxelearth.org/). State-of-the-art engine side: GPU voxelization of textured 3D Tiles, streamed as chunks on demand. A reference for scaling beyond static `.schem` exports.

**Consolidated lessons (recurring across all of the above):** (1) RGB -> constrained ~16-block palette quantization; (2) thin features vanish unless specially handled; (3) block budget vs. voxel size dominates design; (4) processing is heavy/long and must be resumable/chunked; (5) source meshes carry noise needing cleanup; (6) blockiness hurts recognizability at coarse resolution. All six map onto IFC: material colors, curtain-wall mullions/glass, model size, runtime, non-watertight solids, and chosen pitch.

## 3. IFC/mesh voxelization techniques & libraries

This is where the strongest *reusable* prior art lives.

**IFC-native:**
- **IfcOpenShell voxelization_toolkit / `voxec`** — [github.com/IfcOpenShell/voxelization_toolkit](https://github.com/IfcOpenShell/voxelization_toolkit) (the `opensourceBIM/...` URL redirects to the *same* repo). MIT C++, ~90 stars, **v0.3.5 (2024-11-27)**, conda-installable. Discretizes IFC into integer-indexed voxel grids (avoiding BRep float-robustness issues), driven by a declarative voxelfile. Supports **per-class grids** (`create_geometry(file, include={"IfcSlab"})`), surface-vs-volume voxelization, boolean/morphological ops (offset, erosion, fill-gaps), connected-components (6-/26-connectivity). The de facto IFC-native voxel engine; it does the hard part but emits OBJ/point-cloud/CSV/JSON, **not** a Minecraft format. Confirmed.
- **IfcVoxNet (Luttun & Krijnen, CIB W78 2024)** — [itc.scix.net/pdfs/w78-2024-paper_68.pdf](https://itc.scix.net/pdfs/w78-2024-paper_68.pdf). Voxelizes IFC into a **class-labelled occupancy grid** (one element class per voxel), resolving overlaps via a **global priority rule** (roughly by element prevalence/size), then a 3D U-Net on 64³ patches. The priority-ordering algorithm is the citable answer to "which class wins when two IFC element boxes share a voxel" — directly relevant to semantic block mapping.
- **Krijnen et al. 2021, thermal analysis via voxelized IFC** — [repository.tudelft.nl/record/...](https://repository.tudelft.nl/record/uuid:98ad4f7e-105d-4511-baa8-57a088eb7c51). End-to-end IFC -> voxel -> analysis on the IfcOpenShell toolkit; emphasizes robustness over precision for imperfect models. Plus a broader research vein on voxelizing IFC for indoor navigation/pathfinding (TU Delft FSS, Diakité & Zlatanova) and fall-hazard detection — all confirming the IFC->grid step as the shared, under-documented hard part.

**Generic mesh voxelization (theory + tools):**
- **trimesh.voxel.creation** — [trimesh.org/trimesh.voxel.creation.html](https://trimesh.org/trimesh.voxel.creation.html). The project's current path. **Critical gotcha:** `fill()` / `proximity.contains` is documented as *"may be meaningless for non-watertight meshes"* — and IFC-derived triangle meshes are very often not watertight, so naive fill silently drops or mis-fills thin/open elements. Use **surface (`subdivide`) voxelization for glass/mullions** and reserve fill for genuinely closed solids (slabs, walls).
- **Schwarz & Seidel 2010, "Fast Parallel Surface and Solid Voxelization on GPUs"** — [michael-schwarz.com/.../vox-siga10.pdf](https://www.michael-schwarz.com/research/publ/files/vox-siga10.pdf). The theory for thin elements. **Conservative (~26-separating)** sets every voxel a triangle touches, so a sub-voxel mullion/pane is never lost; **6-separating** gives a thinner gap-free shell that is a better seed for flood-fill solidification; the **sparse octree solid** mode is the building-scale storage answer. This is the framework for choosing voxelization "thickness" per element type.
- **cuda_voxelizer (Forceflow)** — [github.com/Forceflow/cuda_voxelizer](https://github.com/Forceflow/cuda_voxelizer). Off-the-shelf GPU Schwarz-Seidel; conservative surface mode preserves thin features; `-solid` needs watertight input (so use surface mode + selective solidify for whole-building input). `.vox`/binvox/morton output.
- **OpenVDB `meshToVolume` + NanoVDB** — [openvdb.org/.../MeshToVolume_8h.html](https://www.openvdb.org/documentation/doxygen/MeshToVolume_8h.html). Sparse narrow-band output (memory scales with surface area, not bounding volume) — the right storage layer for a million-cell building. `UNSIGNED_DISTANCE_FIELD` does **not** require a closed mesh, so open thin elements aren't dropped.
- **topskychen/voxelizer** — [github.com/topskychen/voxelizer](https://github.com/topskychen/voxelizer). CPU alternative whose flood-fill solid mode explicitly works **without watertight input** — a real advantage over trimesh `.fill()`/binvox parity for messy IFC.
- **binvox + viewvox** — [patrickmin.com/binvox](https://www.patrickmin.com/binvox/). Battle-tested; `-e` exact mode is conservative and needs no watertight input. *Confidence note:* the finding's **v1.39 / 2025-11-16** appears fabricated (page shows v1.38, 2025-05-28), and Minecraft `.schematic` output is **binvox's** `-e` mode, **not** `viewvox`. Also: 512³ free cap and X11/XQuartz friction on Apple Silicon.

## 4. Mesh->schematic tooling landscape (2026)

If you ever route IFC -> OBJ/glTF instead of voxelizing in-Python, these are the converters. The project's current `mcschematic`/`litemapy` choice is well-judged.

**Python (headless, what this project already uses):**
- **mcschematic** — [pypi.org/project/mcschematic](https://pypi.org/project/mcschematic/). **v11.4.4 (2026-02-07)**, Python 3.9/3.10, writes Sponge `.schem`, full block-states/NBT via `setBlock`. **No** auto color matching (you supply block IDs — which is what a semantic mapper wants). Confirmed active.
- **litemapy** — [pypi.org/project/litemapy](https://pypi.org/project/litemapy/). **v0.11.0b0 (2025-06-14)**, pure Python, **read+write** `.litematic`. The right lib for the modern Litematica format (supports new blocks; legacy `.schematic` does not).
- **amulet-core** — [pypi.org/project/amulet-core](https://pypi.org/project/amulet-core/). **v1.9.41 (2026-06-17)**, Python 3.11+, numpy. Use only if you need to write blocks **straight into a world save** (not a clipboard); strong cross-version translation. Weaker `.schem`/`.litematic` interchange (Amulet does **not** import `.litematic`).

**OBJ/mesh front-ends (only if you go mesh-route):**
- **ObjToSchematic** — v2 web app at [objtoschematic.com](https://objtoschematic.com/) does voxelization **and** texture/material->block matching (Block Atlas, dithering, smart averaging), exporting `.litematic`/`.schem`/`.nbt`/datapack. **But v2 is GUI/web-only** (hard to automate). The **v1 repo** [github.com/LucasDower/ObjToSchematic](https://github.com/LucasDower/ObjToSchematic) is **archived/legacy** but is the only variant with a documented headless path — *confidence note:* the exact recipe (`tools/headless-config.ts` + `npm run headless`) was not visible in the fetched README; verify before relying on it.
- **MeshToVox / FileToVox (Zarbuz)** — MeshToVox voxelizes OBJ/FBX/glTF/STL with textures to colored `.vox`; FileToVox no longer reads OBJ (regression). .NET; two-step and Mac support under-documented.
- **drububu** ([drububu.com/.../voxelizer](https://drububu.com/miscellaneous/voxelizer/)) and **Bloxelizer** ([bloxelizer.com](https://bloxelizer.com/converter)) — zero-install browser OBJ/glTF->schematic; web-only, drububu emits **legacy** `.schematic`, non-commercial license caveat.

**Color/semantic mapping techniques worth stealing:**
- **TwentyFiveSoftware/voxelizer** — [github.com/TwentyFiveSoftware/voxelizer](https://github.com/TwentyFiveSoftware/voxelizer). Documented **sRGB -> CIE-Lab + delta-E** matching, plus a **separate stained-glass palette for transparent materials** — exactly the glazing heuristic an IfcWindow mapper wants. (Tiny repo; use as technique, not product.)
- **IfcOpenShell surface styles** — [blog.ifcopenshell.org/2014/02/surface-styles.html](http://blog.ifcopenshell.org/2014/02/surface-styles.html). Authoritative per-element RGB lives in `IfcStyledItem -> IfcSurfaceStyle -> IfcSurfaceStyleShading -> IfcColourRgb`, **not** `IfcMaterial` — and styled items can override material color. This is the correct front-end for any color->block step.
- **Minecraft Wiki color palette** — [minecraft.wiki/w/Tutorial:Color_palette](https://minecraft.wiki/w/Tutorial:Color_palette). Ground-truth block-color reference for building per-category sub-palettes (concrete/terracotta for walls; glass for glazing).

**Format targets:** prefer **Sponge `.schem` v3** ([github.com/SpongePowered/Schematic-Specification](https://github.com/SpongePowered/Schematic-Specification)) and/or **`.litematic`**; avoid legacy MCEdit `.schematic` (no DataVersion / new-block support).

## 5. Import-into-Minecraft logistics & limits

For ~7500 walls + ~20000 curtain-wall members across 13 storeys, the import path — not the voxelizer — is the likely bottleneck.

- **WorldEdit** — [worldedit.enginehub.org/.../clipboard](https://worldedit.enginehub.org/en/latest/usage/clipboard/). `//schem load` + `//paste` (with `-a` skip-air, `-o` original-origin for deterministic placement). **Holds the whole clipboard in RAM** — the binding constraint for large models.
- **FastAsyncWorldEdit (FAWE)** — [modrinth.com/plugin/fastasyncworldedit](https://modrinth.com/plugin/fastasyncworldedit). The single most important tool for million-block pastes: **disk-backed clipboard + streaming NBT** (fixed memory overhead), async edits. Practical guidance: throttle with `//fast` then `//limit` (ramp 100k -> 300k), keep the executor **present** (leaving mid-paste can crash the JVM — [issue #1985](https://github.com/IntellectualSites/FastAsyncWorldEdit/issues/1985)). Users report ~4M-block schematics working with adequate RAM.
- **Litematica (+ Printer)** — [github.com/maruohon/litematica/wiki/Schematic-Pasting](https://github.com/maruohon/litematica/wiki/Schematic-Pasting). Client-side alternative; per-chunk multi-tick paste is built-in chunking. Printer is survival-only and slow — not for bulk import.
- **Amulet Map Editor** — [amuletmc.com](https://www.amuletmc.com/). Offline, no-server, edits region files on disk; cross-edition (Java/Bedrock). Still desktop-memory-bound for very large imports.
- **Vanilla structure blocks / `.nbt`** — [minecraft.wiki/w/Structure_Block](https://minecraft.fandom.com/wiki/Structure_Block). Hard **48×48×48** cap on Java (the load-bearing vanilla limit). **Structure Block Unlimiter** (NeoForge) raises it to a configurable 512³. *Confidence note:* the Bedrock cap is now **64×384×64**, not the "64×255×64" in the findings; the Bedrock NBT size-hack is **dead since 1.20.50**.
- **Build-height envelope** — since 1.18, Java runs **Y -64 to 319 (384 blocks)**. A 13-storey building at ~3.5 m/storey is ~45 m, so vertical headroom is a non-issue at 1 block = 1 m; choose a paste base Y so the top clears 319.

## 6. Key takeaways for THIS project

**What is genuinely novel here.** No confirmed prior art is an **open, scriptable, IFC-native, Python pipeline that voxelizes a full real-world building and emits modern `.schem`/`.litematic`**. The direct precedents are commercial (FME), authoring-tool-bound (Revit octree, VASA), or educational (BeIMCraft). The geo precedents work from GIS/terrain/reality-mesh, not IFC element semantics. The IFC-native voxelizers (IfcOpenShell toolkit, IfcVoxNet, Krijnen thermal) stop at analysis grids and **never emit a Minecraft format**. So the novel contribution is the *last mile* — IFC semantics/materials -> Minecraft blocks -> paste-able schematic at building scale — plus correctly handling this model's specific stressors (mm units, ~20000 thin curtain-wall members).

**Reuse vs. build:**
- **Reuse (proven, drop-in):** `ifcopenshell.geom.iterator` (multithreaded, caches repeated windows/mullions — essential for 20k curtain-wall members) over `create_shape`; `mcschematic`/`litemapy` writers (already chosen — good); the IfcOpenShell **surface-style chain** for authoritative per-element RGB; IfcVoxNet's **priority-ordering** for overlapping element classes; TwentyFiveSoftware's **CIE-Lab/delta-E + separate glass palette** for color matching.
- **Build (the genuinely new glue):** the semantic IFC-class/material -> block-palette map; the thin-element handling policy; and chunked/resumable output. Optionally evaluate the **IfcOpenShell voxelization_toolkit** as a more robust integer-math replacement for trimesh if non-watertight fill becomes a problem.
- **Don't bother:** binvox (X11/Apple-Silicon friction, 512³ cap, monochrome), FileToVox (dropped OBJ input), ObjToSchematic v2 (no CLI), Litematica Printer (survival-only). The mesh-export detour (IFC->OBJ->voxelizer) adds a lossy hop and discards IFC semantics — staying IfcOpenShell-native in Python is the better architecture.

**Top 5 practical recommendations:**
1. **Voxelize thin elements with conservative/surface voxelization, not fill.** trimesh `.fill()` is documented as meaningless for non-watertight meshes, and IFC solids usually aren't watertight; with ~20000 curtain-wall members at mm precision, naive fill will silently delete glass and mullions. Use surface (`subdivide`) voxelization (the conservative/26-separating idea from Schwarz & Seidel) for glazing/framing and reserve fill for closed solids (slabs, walls). This is the single highest-risk correctness issue.
2. **Confirm units explicitly: IfcOpenShell outputs meters by default, regardless of the mm file.** `convert-back-units` defaults False, so a 3000 mm wall comes out as 3.0. Pick your scale (likely 1 block = 1 m, or sub-divide) off the meter values; if you need file units, set `convert-back-units=True` or compute `ifcopenshell.util.unit.calculate_unit_scale`. This is a confirmed, common pitfall.
3. **Keep IFC class/material per voxel and map semantically — don't go through color alone.** Voxelize per IfcClass (the toolkit pattern), resolve voxel-class collisions with a priority rule (IfcVoxNet), pull RGB from the surface-style chain, and use per-category sub-palettes (concrete/terracotta for walls, **stained glass for windows/curtain walls**). This preserves intent better than RGB-nearest matching and directly addresses the palette-quantization pitfall every geo precedent reports.
4. **Plan the import path around FAWE from the start, and emit tiled output.** Vanilla WorldEdit OOMs on large clipboards; target FAWE's disk-backed clipboard, document the `//fast` + `//limit` ramp, and split the building (e.g., per-storey — you have 13) into region schematics that paste at fixed offsets. This also gives natural **resumability**, which mesh_to_schematic and the national builds all flag as essential for heavy runs.
5. **Validate by round-trip and budget your blocks.** Acceptance test = the generated `.schem` v3 actually loads and pastes in WorldEdit/FAWE (and re-export via Mineways/Amulet to sanity-check geometry). Separately, profile block count vs. pitch *before* a full run: greedy-meshing/RLE-style compression of the grid was a gap across all prior art and directly determines whether the output is paste-able. Use the multithreaded iterator with `include=` filters and stream/discard each shape in the loop ([memory issue #6905](https://github.com/IfcOpenShell/IfcOpenShell/issues/6905)) so an 80 MB model doesn't exhaust RAM.

**Honesty on confidence:** Section 1–2 named projects and Section 3–5 core libraries/limits are confirmed (fetched). Lower-confidence items, flagged inline: binvox version and the binvox-vs-viewvox `.schematic` attribution (likely wrong), OS source-data/2013-v1 details (true but not at the cited URL), Bedrock structure cap (now 64×384×64), ObjToSchematic v1 headless recipe (plausible, unverified), and FME's exact attribute-band layout (workflow real, specifics unverified). The van Treeck/RWTH angle has **no confirmed voxelization artifact** — do not cite it as a voxelization precedent.

## Gaps to note

Genuinely relevant prior art / considerations that none of the angles surfaced:

- **Bonsai (formerly BlenderBIM)** — the flagship open-source IfcOpenShell-based IFC authoring/geometry app in Blender. It's the obvious open companion to IfcConvert for **IFC→OBJ/glTF with materials**, and Blender then feeds any mesh voxelizer. Surprising omission given how much of the pipeline is IfcOpenShell-based.
- **web-ifc / IFC.js / "That Open Engine"** — the WASM IFC geometry engine for JS/Three.js. Relevant for a browser-side IFC→mesh→voxel route (pairs with web voxelizers like ObjToSchematic 2 / Bloxelizer); no in-browser IFC geometry path was covered.
- **IFC georeferencing** — `IfcMapConversion` / `IfcProjectedCRS` / IfcSite true-north & elevation. The import-logistics angle covers Y-range and paste offsets, but nothing covers extracting real-world CRS from IFC to place the build at deterministic world coordinates (IfcConvert `--center-model`/`--model-offset` is the closest, and only partial).
- **Direct region/anvil NBT writers** — Only Amulet (world saves) and schematic libs were covered. Libraries that write `.mca` region files directly (e.g. `anvil-parser`, `Querz/NBT`, `nbtlib`) are an alternative to the schematic→WorldEdit/FAWE paste path for very large models and weren't mentioned.
- **Minecraft EULA / commercial-use licensing** — For a research-or-commercial BIM deliverable, Mojang's brand/commercial-use guidelines (and Bedrock Marketplace rules) are a real constraint; no finding flags the legal dimension.
- **Greedy meshing / block-budget optimization** — "Block budget" appears as a *resolution* tradeoff, but there's no mention of run-length/greedy-meshing or palette-RLE to compress the output grid, which directly affects whether a million-block schematic is paste-able.
